LLM Training Optimization Summary

Architecture & Parameters

The final model is a custom GPT architecture strictly capped under 2,000,000 parameters. The optimal configuration was determined to be roughly ~1.1M parameters to balance learning capacity with local Windows CPU execution constraints.

Parameter Value Rationale
Vocab Size 1024 Custom BPE compression minimizes Hindi UTF-8 byte fragmentation.
Block Size 128 Restricts attention overhead to avoid severe computation slowdowns.
Layers / Heads 5 / 4 Provides sufficient depth for reasoning while avoiding CPU thread-thrashing.
Embedding Dim 128 Keeps parameter count lean; weights are tied to the LM head to save budget.

Runlog & Notes Summary

Initial Run: The baseline model utilized a standard 256 byte-level tokenizer. While it successfully established a baseline of 2.32 Bits-Per-Byte (bpb), the context window was highly inefficient at processing multi-byte Hindi characters.

Optimized Run: We deployed a BPE tokenizer (vocab 1,024), effectively compressing the corpus by 46%. This allowed the 128-token context window to view significantly more semantic data per step. Severe bottlenecks were encountered with Windows PyTorch execution (silent multiprocessing deadlocks and fp16 software emulation). We bypassed these by utilizing manual tensor slicing in place of DataLoader, capping torch.set_num_threads(4), and running native fp32 math.

Metric Optimization

Bits-Per-Byte (bpb) was optimized primarily through data compression rather than raw scaling. By reducing the number of tokens required to represent the same amount of text, the model intrinsically predicts fewer sequential steps per concept. This allowed us to keep the context window small (128) to bypass quadratic attention penalties, whilst still achieving a highly competitive bpb score within the 2,000-step constraint.

Human vs. Machine Effort Allocation

Human Contribution:

  • Strategic Direction: Defined the architectural boundaries, parameter caps, and optimization goals.
  • Constraint Management: Identified Windows-specific hardware bottlenecks (CPU thrashing, `spawn` vs `fork` multiprocessing deadlocks, emulation traps) and dictated the necessary workarounds.
  • Debugging Logic: Diagnosed Time Limit Exceeded (TLE) errors in the tokenizer and terminal buffering issues during the training loop.

Machine Contribution (Coding Agent):

  • Code Generation: Authored the pure-Python standard library BPE tokenizer implementation.
  • Algorithmic Optimization: Wrote the regex-based word caching logic to reduce $O(N^2)$ execution times to $O(1)$ lookups.
  • Boilerplate & Formatting: Generated the PyTorch training loops, implemented the Cosine Learning Rate scheduler, and formatted this HTML summary and Markdown documentation.
